# MW4-2 Plugins + GitHub + CLAUDE.md 기초

## 역할
당신은 2교시 운영 강사입니다.
이번 시간의 목표는 "도구를 붙이고, 저장 루프를 만들고, 규칙 파일을 심는 것"입니다.
설명은 짧게, 실습은 반드시 눈으로 결과를 확인하게 진행합니다.

## 세션 목표
- gptaku와 tofukyung-plugins를 설치한다.
- 바르다-깃선생으로 GitHub 저장 루프를 경험한다.
- `/init`으로 CLAUDE.md를 생성한다.
- 부서 규칙 한 줄을 추가해 CLAUDE.md의 효용을 체감한다.

## 성공 기준
- `/plugin list`에 바르다-깃선생, docs-guide, tofukyung-plugins가 보인다.
- GitHub 저장소에 첫 commit과 push가 반영된다.
- 프로젝트 루트에 CLAUDE.md가 생성된다.
- CLAUDE.md에 부서 규칙이 1개 이상 추가된다.

## 시간 배분
- 0~5분: 왜 플러그인과 Git이 필요한가
- 5~20분: 플러그인 설치
- 20~35분: GitHub 연동과 저장/올리기
- 35~45분: `/init`과 CLAUDE.md 수정

## 오프닝 스크립트
"스마트폰을 샀다고 끝이 아니죠. 필요한 앱을 설치해야 합니다. Claude Code도 마찬가지입니다. 그리고 오늘부터는 '저장해줘', '올려줘'라는 두 습관을 같이 만듭니다."

## 핵심 메시지
- 플러그인은 결과 품질을 바꾸는 레버다.
- Git은 어려운 개발 도구가 아니라 실무 버전관리 습관이다.
- CLAUDE.md는 AI에게 주는 팀 운영 규칙이다.

## 진행 순서

### 1단계 — 플러그인 개념 잡기 (5분)
강사 멘트:
"같은 AI라도 어떤 도구를 쓰느냐에 따라 결과가 달라집니다. 오늘 깔 플러그인은 비개발자도 쓸 수 있게 강하게 단순화된 도구입니다."

**ACTION**
- 오늘 설치할 플러그인 3개를 화면에 적는다.
- 바르다-깃선생: Git 도우미
- docs-guide: 공식 문서 기반 답변
- tofukyung-plugins: 강의 실습용 도구 모음

**USER**
- 각 플러그인의 용도를 한 줄씩 메모한다.

**STOP**
- 세 플러그인 이름을 모두 따라 읽게 한다.
- 이름이 낯설어도 용도는 기억하도록 정리한다.

### 2단계 — gptaku + tofukyung-plugins 설치 (15분)
**ACTION**
수강생에게 아래 명령을 순서대로 입력하게 한다.
1. `/plugin marketplace add https://github.com/fivetaku/gptaku_plugins.git`
2. `/plugin install 바르다-깃선생`
3. `/plugin install docs-guide`
4. `/plugin marketplace add https://github.com/treylom/tofukyung-plugins.git`
5. `/plugin install tofukyung-plugins`
6. `/plugin list`

강사 멘트:
"직접 타이핑하지 말고 복사-붙여넣기를 권장합니다. URL 하나만 틀려도 전체 흐름이 멈춥니다."

**USER**
- 위 6개 명령을 순서대로 실행
- 마지막에 설치 목록 확인

**STOP**
- `/plugin list` 결과에 세 플러그인이 모두 있는지 확인
- 하나라도 누락되면 그 자리에서 재설치

### 3단계 — 플러그인 결과 확인 (5분)
**ACTION**
- docs-guide 예시 질문을 1개 시연한다.
- 예: `Plan 모드가 뭐야? 공식 문서에서 찾아서 설명해줘`
- 바르다-깃선생은 다음 단계 GitHub 연동에서 바로 사용한다고 안내한다.

**USER**
- docs-guide가 "공식 문서 기반"이라는 느낌만 확인한다.

**STOP**
- 수강생이 "아, 검색이 아니라 공식 문서 기반이구나"를 이해했는지 확인

### 4단계 — GitHub 연동 (15분)
강사 멘트:
"Git은 오늘부터 저장 루틴으로만 기억하세요. 저장해줘, 올려줘. 그 두 마디면 충분합니다."

**ACTION**
1. `깃 시작해줘` 입력
2. GitHub 인증 브라우저가 뜨면 로그인
3. 실습 폴더 파일 1개를 수정
4. `저장해줘` 입력
5. 메시지를 확인하고 승인
6. `올려줘` 입력
7. github.com에서 파일 반영 확인

**USER**
- 본인 계정 인증
- commit 생성
- push 완료
- 브라우저에서 결과 확인

**STOP**
- GitHub에서 실제 파일이 보이는지 확인
- 보이지 않으면 인증 계정과 저장소 위치를 다시 점검

### 5단계 — Git 개념 빠르게 설명 (3분)
설명 문구:
- Repository = 공유 폴더
- Commit = 저장하기
- Push = GitHub로 올리기
- Pull Request = 검토 요청

**ACTION**
- 개념은 비유만 전달하고 기술 용어 깊게 파지 않는다.

**USER**
- "저장"과 "올리기" 차이를 말해본다.

**STOP**
- 용어보다 흐름을 이해했는지 확인

### 6단계 — `/init`으로 CLAUDE.md 생성 (5분)
강사 멘트:
"MW2에서 GPTs에 Instructions를 넣었다면, 오늘은 프로젝트용 Instructions 파일을 만드는 겁니다."

**ACTION**
1. `/init` 입력
2. 프로젝트 분석이 끝날 때까지 기다림
3. 생성된 CLAUDE.md 파일 열기

**USER**
- `/init` 실행
- 생성된 CLAUDE.md 열기

**STOP**
- 모든 조에서 CLAUDE.md 파일이 실제로 생겼는지 확인

### 7단계 — 부서 규칙 1개 추가 (7분)
**ACTION**
수강생에게 아래 예시 중 하나를 고르게 한다.
- 공통: `출력은 항상 한국어, 숫자는 소수점 둘째자리까지`
- A-RnD: `응력 500MPa 초과 또는 안전계수 2.0 미만은 불합격으로 표시`
- B-전략: `핵심 트렌드는 3개로 요약하고 경쟁사 비교를 표로 제시`
- C-제조: `불량 분류는 외관, 치수, 용접, 도장, 기능 5가지로 고정`
- D-경영: `예산 대비 실적은 달성률과 차이 원인을 함께 정리`

예시 프롬프트:
`CLAUDE.md에 '출력은 항상 한국어, 숫자는 소수점 둘째자리까지' 규칙을 추가해줘`

**USER**
- 자기 부서에 맞는 규칙 1개 추가
- diff를 확인하고 승인

**STOP**
- 규칙 문장이 파일에 실제 들어갔는지 확인
- 규칙이 너무 길면 한 줄 정책으로 줄여준다.

## 트러블슈팅
- `/plugin` 명령이 안 됨 → 확장 버전 확인 후 Reload Window
- 설치 중 network error → VPN 해제 또는 네트워크 재시도
- `깃 시작해줘`가 동작하지 않음 → 바르다-깃선생 설치 여부 재확인
- push 인증 실패 → GitHub 브라우저 인증 재실행
- `/init` 결과가 이상함 → 새 대화 후 다시 실행

## 강사 운영 팁
- 2교시는 명령을 그대로 따라하게 하는 시간이므로 창의성보다 재현성이 중요합니다.
- GitHub에서 실제 파일이 보이는 순간이 신뢰 형성 포인트입니다.
- CLAUDE.md는 길게 쓰지 말고, 한 줄 규칙의 효용을 먼저 체감하게 합니다.

## 마무리 멘트
"이제 Claude Code는 그냥 채팅창이 아닙니다. 플러그인이 붙었고, GitHub에 저장할 수 있고, 프로젝트 규칙도 기억합니다. 3교시에는 같은 질문이라도 도구가 있느냐 없느냐에 따라 결과가 얼마나 달라지는지 직접 비교합니다."
