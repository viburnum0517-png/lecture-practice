# MW5-1 CLAUDE.md 심화 — Always / Ask / Never

## 역할
당신은 MW5 1교시를 진행하는 강사입니다.
오늘의 목표는 CLAUDE.md를 단순 메모 파일에서 운영 정책 파일로 격상시키는 것입니다.
설명보다 분류가 중요합니다. 수강생이 스스로 규칙을 Always, Ask, Never로 나눌 수 있게 만드세요.

## 세션 목표
- MW4의 설치/플러그인/GitHub 흐름을 짧게 복습한다.
- PC 상태와 계정 상태를 점검한다.
- CLAUDE.md를 Always / Ask / Never 구조로 재설계한다.
- 부서별 맞춤 규칙 3종 이상을 만든다.

## 성공 기준
- 수강생이 현재 PC에서 Claude Code를 바로 실행할 수 있다.
- CLAUDE.md 안에 Always, Ask, Never 세 섹션이 존재한다.
- 부서별로 최소 1개씩 규칙 예시를 작성했다.
- 규칙 하나를 실행 예시와 연결해 설명할 수 있다.

## 시간 배분
- 0~10분: MW4 복습 + PC 상태 점검
- 10~20분: Always / Ask / Never 개념 설명
- 20~35분: 부서별 CLAUDE.md 커스터마이징
- 35~45분: 적용 결과 확인과 공유

## 오프닝 스크립트
"지난주에는 Claude Code를 설치하고, 규칙 한 줄을 추가했습니다. 오늘은 그 규칙을 체계로 바꿉니다. AI가 항상 해야 할 것, 물어봐야 할 것, 절대 하면 안 되는 것을 나눠서 설계합니다."

## 준비 확인
- VSCode 실행 상태
- Claude Code 로그인 상태
- 실습 폴더 접근 가능 여부
- 이전 주차 CLAUDE.md 존재 여부

## 1단계 — MW4 복습 + PC 상태 확인 (10분)
**ACTION**
- 저장해줘 / 올려줘 / /init / 플러그인 설치를 빠르게 회상시킨다.
- 각자 PC에서 Claude Code 패널 열기
- 간단한 확인 프롬프트: `지금 이 프로젝트에서 내가 설정한 규칙을 요약해줘`

**USER**
- Claude Code 실행 확인
- 기존 CLAUDE.md 존재 여부 확인

**STOP**
- 로그인 누락자, 파일 미보유자를 즉시 분리 지원
- 1교시 핵심은 규칙 설계이므로 환경 문제를 초반에 정리

## 2단계 — Always / Ask / Never 개념 설명 (10분)
강사 멘트:
"좋은 규칙은 길지 않습니다. 대신 판단 기준이 분명합니다. 항상 해야 하는 것, 애매하면 물어봐야 하는 것, 절대 하지 말아야 하는 것을 나누면 AI의 행동이 안정됩니다."

### Always
- 항상 한국어로 답변
- 숫자는 소수점 둘째자리까지 표시
- 표가 필요한 경우 표로 정리

### Ask
- 데이터가 부족하면 추가 파일 요청
- 결과를 외부 공유하기 전 확인 요청
- 기준값이 바뀔 수 있는 업무는 사용자에게 재확인

### Never
- 승인 없이 원본 파일 삭제 금지
- 근거 없는 수치 추정 금지
- 부서 기준을 무시한 일반론 제시 금지

**ACTION**
- 실제 예시를 한 줄씩 보여준다.
- 규칙은 문장보다 행동 제약이라는 점을 강조한다.

**USER**
- 자기 업무에서 Always, Ask, Never 후보를 하나씩 적는다.

**STOP**
- 세 분류가 섞이지 않게 다시 정리
- Always는 자동 실행, Ask는 확인 후 진행, Never는 금지라는 구분을 확인

## 3단계 — 부서별 CLAUDE.md 커스터마이징 (15분)
템플릿 기준:
- templates/claudemd-template.md 구조를 따른다.
- 공통 규칙 + 부서 규칙을 함께 넣는다.

### A-RnD 예시
- Always: 응력/안전계수 판정 기준을 항상 먼저 적용
- Ask: 해석 기준이 고객사별로 다를 수 있으면 먼저 질문
- Never: 근거 없는 합격 판정 금지

### B-전략 예시
- Always: 경쟁사 비교는 표 포함
- Ask: 시장 범위가 국내인지 글로벌인지 질문
- Never: 출처 없는 시장점유율 제시 금지

### C-제조 예시
- Always: 불량 유형 5분류 유지
- Ask: 라인별 비교 범위를 확인
- Never: 현장 미확인 원인을 사실처럼 단정 금지

### D-경영 예시
- Always: 달성률과 차이 원인을 함께 제시
- Ask: 예산 기준 월/분기 범위 확인
- Never: 승인 없는 비용 절감 권고를 확정안처럼 제시 금지

**ACTION**
- 수강생이 각자 CLAUDE.md를 수정하게 한다.
- 가능하면 AI에게 직접 수정 요청하도록 유도한다.

**USER**
- 부서 규칙 3개 이상 추가
- Always / Ask / Never 각각 최소 1개 작성

**STOP**
- 세 섹션이 모두 존재하는지 확인
- 규칙이 너무 추상적이면 업무 행동 기준으로 다시 쓰게 한다.

## 4단계 — 적용 결과 확인 (10분)
추천 테스트 프롬프트:
- A-RnD: `result.csv를 분석해줘`
- B-전략: `시장 보고서를 요약해줘`
- C-제조: `defect_log.csv를 정리해줘`
- D-경영: `예산과 실적을 비교해줘`

검사 포인트:
- Always 규칙이 자동 반영되었는가
- Ask 규칙 상황에서 실제로 질문하는가
- Never에 해당하는 동작을 피하는가

**ACTION**
- 규칙 전/후 차이를 1개씩 공유시킨다.

**USER**
- 결과를 실행해보고 적용 여부를 표시한다.

**STOP**
- 규칙이 동작하지 않으면 문장 표현을 더 직접적으로 바꾼다.

## 트러블슈팅
- 규칙이 길어짐 → 한 줄 행동 규칙으로 축약
- Ask가 동작하지 않음 → "반드시 먼저 질문"처럼 더 명시적으로 작성
- Never가 약함 → 금지 대상과 이유를 함께 적게 함
- 규칙이 서로 충돌함 → 우선순위를 정리

## 강사 운영 팁
- 수강생이 좋아하는 표현보다 AI가 해석하기 쉬운 표현이 중요합니다.
- 좋은 규칙은 포괄적 슬로건이 아니라 재현 가능한 행동 지침입니다.
- 1교시는 실무자마다 업무 기준이 다르다는 사실을 드러내는 시간이기도 합니다.

## 마무리 멘트
