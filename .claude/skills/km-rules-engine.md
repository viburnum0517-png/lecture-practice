# Rules Engine Skill

규칙서 설계 → DA 검증 → 동결 → 스크립트 자동 생성의 재사용 가능 패턴.
대규모 배치 작업 전 일관성을 보장하기 위한 사전 설계 도구.

## 적용 시점

- `km-archive-reorganization.md` Phase R3에서 호출
- 독립적으로도 사용 가능 (규칙 기반 배치 작업 필요 시)

---

## Step 1: 데이터 수집

### 입력 소스

| 소스 | 설명 |
|------|------|
| analyst 결과 | `.team-os/artifacts/analyst-*-result.txt` |
| vault 탐색 결과 | Explore 에이전트 보고서 |
| CATEGORY_DESIGN.md | 카테고리 구조 (Phase R2 산출물) |
| 사용자 요구사항 | AskUserQuestion 응답 |

### 데이터 종합

```
1. 모든 analyst 결과 읽기
2. topic_clusters 병합 (중복 제거)
3. series 병합 (교차 검증)
4. cross_links 교차 카테고리 맵 생성
5. hubs 허브 노트 후보 정리
```

## Step 2: 규칙서 설계

### 규칙서 템플릿

```markdown
# {TARGET} 규칙서

> {간략 설명}. {분석 근거}.

---

## 1. Footer 템플릿

```markdown
---

## 관련 노트

### 같은 시리즈
- [[시리즈 글 제목]] — n/N편

### 유사 주제
- [[관련 글 제목]] — 연결 근거 키워드

### 답글/원글
- 원글: [[원글 제목]]
- 답글: [[답글 제목]]

---
← [[카테고리-MOC|카테고리명]] · [[메인-MOC|전체 목록]]
` ` `

### 적용 규칙

| 섹션 | 조건 | 비고 |
|------|------|------|
| 같은 시리즈 | 시리즈 멤버일 때만 | 양방향 필수, 순서 표기 |
| 유사 주제 | 같은 클러스터 + 교차 카테고리 | 최대 5개 |
| 답글/원글 | 답글 관계 존재 시 | 원글 부재 시 "외부 글" |
| 빈 섹션 | 해당 없으면 섹션 자체 생략 | 빈 ### 헤더 금지 |

---

## 2. 시리즈 레지스트리

| # | 시리즈명 | 카테고리 | 멤버 수 | 상태 |
|---|---------|---------|--------|------|
| ... | ... | ... | ... | ... |

### 시리즈 멤버 정의 규칙

- 본편만 포함: 시리즈 제목을 직접 표방한 글만 카운트
- 메타글 별도: 출판 후기, 연재 예고는 '유사 주제'에서 연결

---

## 3. 교차 카테고리 링크 맵

| 교차 주제 | 카테고리 A | 카테고리 B | 연결 방식 |
|----------|-----------|-----------|----------|
| ... | ... | ... | 양방향 유사주제 |

---

## 4. 허브 노트

| 허브 노트 | 연결 가능 노트 수 | 주요 연결 |
|----------|-----------------|----------|
| ... | ... | ... |

---

## 5. MOC 계획

| MOC | 구조 |
|-----|------|
| 메인 MOC | 카테고리 테이블 + 시리즈 인덱스 + 통계 |
| 카테고리 MOC | 소속 글 목록 + 시리즈 + 교차 링크 |
```

## Step 3: DA 제출

### 제출 패키지

```
DA에게 다음을 함께 전달:

1. 규칙서 전문 ({TARGET}_RULES.md)
2. 3개 예시 적용 결과:
   - 시리즈 멤버 1개 (푸터에 '같은 시리즈' 포함)
   - 교차 카테고리 1개 (푸터에 '유사 주제' 교차 포함)
   - 독립 글 1개 (유사 주제만 있는 최소 푸터)
3. 검토 요청 사항:
   - 규칙 일관성
   - 빈틈/모호함
   - 적용 시 예외 케이스
```

### DA 메시지 형식 (AT 모드)

```
SendMessage(
  recipient: "devils-advocate",
  content: """
  규칙서를 검토해주세요:

  {규칙서 전문}

  예시 적용 결과:
  {3개 예시}

  검토 관점:
  1. 규칙 일관성 및 모호함
  2. 예외 케이스 누락
  3. 적용 시 충돌 가능성

  recommendation: ACCEPTABLE 또는 CONCERN + 구체적 이슈 목록
  """,
  summary: "Rules review request"
)
```

---

## Step 4: CONCERN 처리

### CONCERN 응답 처리

```
DA CONCERN 수신 시:

1. 이슈 목록 추출
2. 각 이슈에 대해:
   a. 규칙서 수정 (이슈가 타당한 경우)
   b. 반론 제시 (이슈가 부당한 경우, 근거 포함)
3. 수정된 규칙서 + 수정 사항 설명 재제출
4. 최대 3회 반복
```

### CONCERN 이슈 유형별 대응

| 이슈 유형 | 대응 |
|----------|------|
| 정의 모호 | 명확한 정의 + 예시 추가 |
| 수량 제한 미정 | 구체적 최대/최소 수치 명시 |
| 예외 케이스 누락 | 예외 규칙 추가 |
| 충돌 가능성 | 우선순위 규칙 추가 |
| 구조적 문제 | 규칙서 구조 재설계 |

---

## Step 5: ACCEPTABLE → 동결

```
DA ACCEPTABLE 수신 시:

1. 규칙서 최종 버전 저장
   → .team-os/artifacts/{TARGET}_RULES.md

2. 규칙서에 동결 마커 추가:
   > ✅ DA ACCEPTABLE — {날짜} — 동결됨. 수정 시 DA 재검증 필요.

3. TEAM_BULLETIN.md에 기록:
   ## [{timestamp}] - Lead
   **Task**: Rules finalized
   **Findings**: {TARGET}_RULES.md DA ACCEPTABLE. 동결됨.
```

---

## Step 6: 스크립트 자동 생성

### 동결된 규칙서 → Python 스크립트

규칙서의 각 구성요소를 Python 스크립트로 변환:

| 규칙서 구성요소 | Python 스크립트 |
|---------------|----------------|
| Footer 템플릿 + 시리즈 레지스트리 | `apply_footers.py` |
| MOC 계획 | `update_mocs.py` |
| 카테고리 분배 규칙 | `restructure.py` |
| frontmatter 규칙 | `fix_frontmatter.py` |

### 스크립트 생성 원칙

```
1. 규칙서의 데이터를 Python dict/list로 변환
2. 규칙서의 조건문을 Python if/else로 변환
3. --dry-run 모드 필수 포함
4. 변경 카운터 출력
5. 에러 시 해당 파일 스킵 + 로그 출력
```

실행은 `km-batch-python.md` 참조.
